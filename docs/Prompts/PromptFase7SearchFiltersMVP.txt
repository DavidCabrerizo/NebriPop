Nombre de la fase:
Fase 7 — Búsqueda y filtros de productos

Repositorio oficial:
https://github.com/DavidCabrerizo/NebriPop/

Contexto:
Ya se han completado las fases anteriores del MVP de NebriPop:

- Fase 1: estructura real del proyecto.
- Fase 2: diseño de la base de datos SQLite.
- Fase 3: backend base con Axum conectado a SQLite.
- Fase 4: frontend base con Leptos conectado al backend Axum.
- Fase 5: integración completa Frontend + Backend + SQLite.
- Fase 5B: gestión de imágenes de productos.
- Fase 6: sistema básico de usuarios.

NebriPop es una aplicación tipo Wallapop para compraventa de productos de segunda mano de todo tipo. Actualmente los usuarios pueden publicar productos, ver productos, consultar detalles, subir imágenes y asociar productos a usuarios. Ahora necesitamos añadir búsqueda y filtros para que el usuario pueda encontrar productos de forma rápida.

Stack tecnológico definitivo:
- Frontend: Rust + Leptos
- Backend: Rust + Axum
- Base de datos: SQLite
- Base de datos real: database/nebripop.db
- Backend local: http://127.0.0.1:3000
- Frontend local: http://127.0.0.1:8081
- Control de versiones: Git + GitHub

Objetivo principal de la Fase 7:
Implementar búsqueda y filtros de productos en NebriPop, permitiendo buscar por texto y filtrar por categoría, precio, estado, ubicación y estado de publicación.

Importante:

## 4. RustBackend

RustBackend será el agente principal de implementación backend de esta fase.

Debe trabajar en:

backend/

Debe modificar el endpoint:

GET /products

para aceptar filtros mediante query params.

Parámetros soportados:

- search
- category_id
- min_price
- max_price
- condition
- location
- status
- sort

Funcionamiento esperado:

### search

Debe buscar coincidencias en:

- title
- description

Debe usar LIKE de SQLite de forma segura y parametrizada.

Ejemplo:

GET /products?search=iphone

### category_id

Debe filtrar por categoría.

Ejemplo:

GET /products?category_id=1

### min_price y max_price

Debe filtrar por rango de precio.

Ejemplo:

GET /products?min_price=50&max_price=300

### condition

Debe filtrar por estado del producto.

Valores permitidos:

- new
- like_new
- good
- used
- damaged

### location

Debe filtrar por ubicación usando búsqueda parcial.

Ejemplo:

GET /products?location=Madrid

### status

Debe filtrar por estado de publicación.

Valores permitidos:

- available
- reserved
- sold

### sort

Debe permitir ordenación básica:

- newest
- price_asc
- price_desc

Si no se envía sort, usar newest por defecto.

Reglas backend obligatorias:

- No construir SQL inseguro concatenando directamente datos del usuario.
- Usar consultas parametrizadas.
- Validar tipos numéricos.
- Rechazar valores inválidos de condition.
- Rechazar valores inválidos de status.
- Rechazar valores inválidos de sort.
- Si no hay resultados, devolver lista vacía.
- Mantener compatibilidad con GET /products sin filtros.
- Mantener main_image_url en la respuesta.
- No romper GET /products/:id.
- No modificar endpoints de usuarios salvo que sea estrictamente necesario.
