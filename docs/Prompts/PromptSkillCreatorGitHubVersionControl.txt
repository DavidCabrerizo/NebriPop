Actúa como Skill Creator dentro de Antigravity utilizando la skill `skill-creator`.

Contexto del proyecto:
Estamos desarrollando NebriPop, una aplicación tipo Wallapop académico para estudiantes. El proyecto utiliza Rust tanto en frontend como en backend, Leptos para frontend, Axum para backend, SQLite como base de datos, Antigravity para coordinación de agentes, OpenCode para implementación técnica y GitHub como repositorio remoto.

Repositorio oficial del proyecto:

https://github.com/DavidCabrerizo/NebriPop/

Queremos crear una nueva skill personalizada para que posteriormente pueda ser utilizada por un agente llamado GitHubAgent.

Nombre de la nueva skill:

github-version-control

Skill utilizada para crearla:

skill-creator

Objetivo principal:
Crear una skill especializada en control de versiones con Git y GitHub para revisar cambios del proyecto, detectar modificaciones relevantes, preparar commits claros, evitar subir archivos sensibles y sincronizar el repositorio oficial de NebriPop únicamente cuando el Agent Manager lo autorice.

Descripción de la skill:
La skill `github-version-control` debe ayudar a gestionar el repositorio GitHub del proyecto NebriPop de forma ordenada, segura y documentada. Su finalidad es que el futuro GitHubAgent pueda analizar el estado del repositorio, revisar cambios, proponer commits profesionales, comprobar el remoto, evitar errores de seguridad y preparar subidas a GitHub bajo supervisión del Agent Manager.

Capacidades principales de la skill:

- Revisar el estado del repositorio con `git status`.
- Revisar diferencias con `git diff`.
- Revisar cambios preparados con `git diff --staged`.
- Verificar la rama actual con `git branch`.
- Revisar últimos commits con `git log --oneline`.
- Comprobar el remoto con `git remote -v`.
- Confirmar que el remoto apunta a:

  https://github.com/DavidCabrerizo/NebriPop/

- Detectar archivos nuevos, modificados o eliminados.
- Identificar cambios relevantes dignos de commit.
- Agrupar cambios por temática.
- Proponer mensajes de commit claros y profesionales.
- Evitar commits demasiado grandes o mezclados.
- Detectar archivos sensibles antes de hacer commit.
- Revisar `.gitignore`.
- Recomendar ramas de trabajo si es necesario.
- Preparar commits solo cuando el Agent Manager lo autorice.
- Preparar push solo cuando el Agent Manager lo autorice.
- Preparar resúmenes para DocumentationAgent.
- Coordinarse con QAAgent, ReviewerAgent, DevopsAgent y SequrityAgent cuando proceda.

Cambios considerados dignos de commit:
La skill debe considerar relevantes los cambios relacionados con:

- Nuevas funcionalidades.
- Corrección de errores.
- Pruebas técnicas.
- Documentación importante.
- Creación o actualización de agentes.
- Creación o actualización de skills.
- Cambios en backend.
- Cambios en frontend.
- Cambios en base de datos.
- Cambios de arquitectura.
- Cambios de QA.
- Cambios de seguridad.
- Cambios de DevOps.
- Cierre de fases del proyecto.
- Avances importantes del MVP.

Archivos que la skill debe evitar subir salvo autorización explícita:

- `.env`
- `.env.local`
- Credenciales.
- Tokens.
- Claves privadas.
- Archivos con contraseñas.
- Bases de datos locales con datos sensibles.
- Carpeta `target/`.
- Archivos temporales.
- Archivos de caché.
- Logs locales innecesarios.
- Capturas temporales.
- Archivos personales del sistema.
- Archivos generados automáticamente sin valor para el proyecto.

Reglas de seguridad obligatorias:

- Nunca hacer commit sin autorización explícita del Agent Manager.
- Nunca hacer push sin autorización explícita del Agent Manager.
- Nunca hacer force push.
- Nunca borrar ramas sin autorización.
- Nunca reescribir historial Git sin autorización.
- Nunca subir `.env`.
- Nunca subir credenciales, tokens o claves privadas.
- Nunca subir `target/`.
- Si se detecta un secreto, detener el proceso y avisar al Agent Manager.
- Si `.gitignore` no protege archivos importantes, proponer corregirlo antes de hacer commit.
- No modificar `docs/Logs.md` directamente salvo autorización explícita; normalmente debe preparar un resumen para DocumentationAgent.

Formato de análisis antes de proponer commit:
Cuando un agente use esta skill para preparar un commit, debe devolver:

1. Rama actual.
2. Estado del repositorio.
3. Archivos nuevos.
4. Archivos modificados.
5. Archivos eliminados.
6. Archivos que no deben subirse.
7. Riesgos detectados.
8. Agrupación recomendada de cambios.
9. Mensaje de commit recomendado.
10. Agentes que deberían revisar antes del commit.
11. Solicitud de autorización al Agent Manager.

Formato de análisis antes de proponer push:
Cuando un agente use esta skill para preparar un push, debe devolver:

1. Rama actual.
2. Commits pendientes de subir.
3. Repositorio remoto configurado.
4. Confirmación de que el remoto apunta a:

   https://github.com/DavidCabrerizo/NebriPop/

5. Riesgos detectados.
6. Confirmación de que no se subirán secretos.
7. Solicitud de autorización al Agent Manager.

Formato recomendado de commits:
La skill debe recomendar mensajes siguiendo este estilo:

- `feat: añade estructura inicial del MVP`
- `feat: implementa prueba pre-MVP con Leptos Axum y SQLite`
- `fix: corrige integración frontend-backend`
- `docs: documenta agentes del proyecto`
- `docs: añade prompt de prueba pre-MVP`
- `test: añade checklist QA de integración`
- `chore: configura repositorio GitHub`
- `refactor: mejora estructura del backend Rust`

Coordinación con futuros agentes:
La skill debe estar preparada para que el futuro GitHubAgent se coordine con:

- Agent Manager / Product Manager Agent para autorizar commits y push.
- DocumentationAgent para registrar cambios en `docs/Logs.md`.
- ReviewerAgent para revisar cambios importantes.
- QAAgent para validar pruebas antes de subir cambios funcionales.
- DevopsAgent para revisar cambios de CI/CD, scripts o despliegue.
- SequrityAgent para revisar secretos o riesgos de seguridad.
- RustBackend para cambios backend.
- RustFrontend para cambios frontend.
- DatabaseAgent para cambios SQLite.
- RustArchitectAgent para cambios arquitectónicos.

Comportamiento esperado:
Cuando se utilice la skill `github-version-control`, debe priorizar:

- Seguridad.
- Trazabilidad.
- Commits claros.
- No subir secretos.
- No mezclar cambios sin relación.
- No hacer push sin autorización.
- Mantener el repositorio limpio.
- Facilitar la documentación posterior de los cambios.

Primera tarea recomendada para un agente que use esta skill:
Realizar una auditoría inicial del repositorio:

1. Comprobar si Git está inicializado.
2. Comprobar si existe remoto.
3. Comprobar si el remoto apunta a:

   https://github.com/DavidCabrerizo/NebriPop/

4. Comprobar rama actual.
5. Revisar archivos modificados.
6. Revisar archivos sin seguimiento.
7. Revisar `.gitignore`.
8. Detectar riesgos.
9. Recomendar siguiente commit.
10. No hacer commit ni push todavía.
