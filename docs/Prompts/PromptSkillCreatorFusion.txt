Contexto del proyecto:
Estamos desarrollando NebriPop, una aplicación tipo Wallapop académico para estudiantes. El proyecto utiliza Rust tanto en frontend como en backend, SQLite como base de datos y una metodología asistida por IA con Antigravity y OpenCode.

Queremos crear una nueva skill que fusione las capacidades de dos skills existentes:

1. `clean-code`
2. `rust-best-practices`

La nueva skill debe combinar todas las cualidades de ambas para que pueda ser utilizada por agentes técnicos del proyecto, especialmente agentes relacionados con arquitectura, backend, frontend, revisión de código y calidad técnica.

CleanRustSkill

CleanRustSkill

Objetivo principal de la skill:
La skill CleanRustSkill debe fusionar las mejores prácticas de Clean Code con las mejores prácticas específicas de Rust, creando una skill especializada en escribir, revisar y mejorar código Rust limpio, mantenible, modular, seguro y fácil de entender.

Descripción de la skill:
CleanRustSkill es una skill especializada en aplicar principios de código limpio y buenas prácticas de Rust dentro de proyectos full-stack. Su objetivo es ayudar a que el código generado para NebriPop sea claro, modular, mantenible, seguro, idiomático y alineado con una arquitectura limpia.

Esta skill debe servir tanto para tareas de implementación como para tareas de revisión técnica, refactorización, validación de calidad y documentación de decisiones de código.

Ámbito de aplicación:
La skill CleanRustSkill se aplicará principalmente sobre:

- Código backend en Rust.
- Código frontend en Rust.
- Servicios.
- Repositorios.
- Handlers.
- Modelos.
- DTOs.
- Componentes frontend.
- Validaciones.
- Gestión de errores.
- Tests.
- Refactorizaciones.
- Revisión de arquitectura.
- Revisión de mantenibilidad.
- Revisión de legibilidad.
- Revisión de duplicidades.
- Revisión de buenas prácticas de Rust.

Cualidades heredadas de la skill clean-code:
La nueva skill debe integrar principios de Clean Code como:

- Código claro y fácil de leer.
- Nombres descriptivos para variables, funciones, módulos y estructuras.
- Funciones pequeñas y con una única responsabilidad.
- Evitar duplicación de código.
- Evitar código muerto.
- Evitar comentarios innecesarios cuando el código puede explicarse por sí mismo.
- Separación clara de responsabilidades.
- Bajo acoplamiento.
- Alta cohesión.
- Organización lógica de archivos y módulos.
- Código fácil de testear.
- Código fácil de mantener.
- Evitar sobreingeniería.
- Priorizar simplicidad.
- Evitar funciones demasiado largas.
- Evitar estructuras demasiado complejas.
- Mantener consistencia de estilo.
- Refactorizar cuando el código se vuelva difícil de entender.
- Documentar decisiones importantes cuando sea necesario.

Cualidades heredadas de la skill rust-best-practices:
La nueva skill debe integrar buenas prácticas específicas de Rust como:

- Uso correcto del sistema de tipos de Rust.
- Uso adecuado de `Result` y `Option`.
- Gestión explícita de errores.
- Evitar `unwrap()` y `expect()` innecesarios en lógica crítica.
- Uso adecuado de ownership y borrowing.
- Evitar clones innecesarios.
- Uso correcto de lifetimes cuando sea necesario.
- Preferir código idiomático de Rust.
- Uso de structs y enums de forma clara.
- Uso de traits cuando aporten valor real.
- Evitar abstracciones innecesarias.
- Uso correcto de módulos.
- Separación entre lógica de dominio, infraestructura y presentación.
- Uso de `serde` para serialización/deserialización cuando proceda.
- Uso correcto de async/await si el backend lo necesita.
- Manejo seguro de concurrencia si aplica.
- Validación clara de entradas.
- Evitar panics en producción.
- Mantener dependencias bajo control.
- Usar `cargo fmt`, `cargo clippy`, `cargo check` y `cargo test`.
- Crear tests unitarios y de integración cuando corresponda.
- Mantener compatibilidad con el stack Rust definido en el proyecto.

Buenas prácticas obligatorias:
La skill debe recomendar siempre que sea posible:

- `cargo check` para verificación rápida.
- `cargo test` para pruebas.
- Tests unitarios para lógica aislada.
- Tests de integración para endpoints o flujos completos.
- Gestión explícita de errores.
- Tipos claros y seguros.
- Separación de responsabilidades.
- Código simple y legible.
- Documentación cuando una decisión técnica no sea evidente.
