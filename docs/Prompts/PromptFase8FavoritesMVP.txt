Actúa como Agent Manager / Product Manager Agent del proyecto NebriPop.

Iniciar la Fase 8 del MVP real de NebriPop.

Nombre de la fase:
Fase 8 — Sistema de favoritos

Repositorio oficial:
https://github.com/DavidCabrerizo/NebriPop/

Contexto:
Ya se han completado las fases anteriores del MVP de NebriPop:

- Fase 1: estructura real del proyecto.
- Fase 2: diseño de la base de datos SQLite.
- Fase 3: backend base con Axum conectado a SQLite.
- Fase 4: frontend base con Leptos conectado al backend Axum.
- Fase 5: integración completa Frontend + Backend + SQLite.
- Fase 5B: gestión de imágenes de productos.
- Fase 6: sistema básico de usuarios.
- Fase 7: búsqueda y filtros de productos.

NebriPop es una aplicación tipo Wallapop para compraventa de productos de segunda mano de todo tipo. Ahora necesitamos implementar el sistema de favoritos para que los usuarios puedan guardar productos que les interesen, quitarlos de favoritos y consultar su lista de productos favoritos.

Stack tecnológico definitivo:
- Frontend: Rust + Leptos
- Backend: Rust + Axum
- Base de datos: SQLite
- Base de datos real: database/nebripop.db
- Backend local: http://127.0.0.1:3000
- Frontend local: http://127.0.0.1:8081
- Control de versiones: Git + GitHub

Objetivo principal de la Fase 8:
Implementar el sistema de favoritos de NebriPop, permitiendo que un usuario pueda añadir productos a favoritos, eliminarlos y consultar su lista de favoritos.

Importante:
Esta fase debe centrarse únicamente en favoritos.

En esta fase SÍ se permite implementar:
- Añadir producto a favoritos.
- Eliminar producto de favoritos.
- Consultar favoritos de un usuario.
- Mostrar estado favorito/no favorito en el listado.
- Mostrar estado favorito/no favorito en el detalle.
- Página o sección de “Mis favoritos”.
- Persistencia en SQLite.
- Validaciones básicas.
- Documentación, QA, seguridad y revisión.

En esta fase NO se debe implementar todavía:
- Mensajes funcionales.
- Chat.
- Pagos.
- Notificaciones.
- Sistema real de compras.
- Recomendaciones inteligentes.
- Geolocalización real.
- Panel de administración.
- Reviews o valoraciones.

Regla crítica de persistencia:

- user_favorites.json
- data.json
- mock.json

## 4. RustBackend

RustBackend será el agente principal de implementación backend de esta fase.

Debe trabajar en:

backend/

Debe implementar endpoints para favoritos.

Endpoints obligatorios:

### GET /users/:id/favorites

Debe devolver la lista de productos favoritos de un usuario.

Debe leer desde SQLite usando la tabla favorites y products.

Respuesta esperada:

[
  {
    "id": 1,
    "user_id": 1,
    "category_id": 1,
    "title": "iPhone 12 usado",
    "description": "iPhone 12 en buen estado",
    "price": 250.0,
    "condition": "used",
    "location": "Madrid",
    "status": "available",
    "main_image_url": "uploads/products/1/imagen_1.jpg",
    "created_at": "2026-05-24T10:00:00",
    "updated_at": "2026-05-24T10:00:00"
  }
]

### POST /favorites

Debe añadir un producto a favoritos.

Body esperado:

{
  "user_id": 1,
  "product_id": 1
}

Validaciones:
- user_id debe existir.
- product_id debe existir.
- No permitir duplicados.
- Si ya existe, devolver mensaje claro.
- No guardar en JSON.
- Guardar en SQLite.

Respuesta esperada:

{
  "message": "Producto añadido a favoritos",
  "user_id": 1,
  "product_id": 1
}

### DELETE /favorites/:product_id?user_id=1

Debe eliminar un producto de favoritos para un usuario.

Ejemplo:

DELETE /favorites/1?user_id=1

Validaciones:
- user_id debe existir.
- product_id debe existir.
- Si no existe el favorito, devolver mensaje claro.
- Eliminar solo el favorito del usuario indicado.

Respuesta esperada:

{
  "message": "Producto eliminado de favoritos",
  "user_id": 1,
  "product_id": 1
}

### GET /favorites/check?user_id=1&product_id=1

Debe indicar si un producto está en favoritos para un usuario.

Respuesta esperada:

{
  "is_favorite": true
}

Reglas backend obligatorias:

- Usar SQLite.
- No usar JSON como persistencia.
- Usar consultas parametrizadas.
- Evitar duplicados mediante UNIQUE(user_id, product_id).
- Validar que user_id y product_id existen.
- No implementar favoritos globales sin usuario.
- No modificar login salvo que sea imprescindible.
- Mantener compatibilidad con GET /products.
- Mantener compatibilidad con GET /products/:id.
