Ya se han completado las fases anteriores del MVP de NebriPop:

NebriPop es una aplicación tipo Wallapop para compraventa de productos de segunda mano de todo tipo. Ahora necesitamos implementar una mensajería básica para que un usuario pueda contactar con el vendedor de un producto.

Stack tecnológico definitivo:
- Frontend: Rust + Leptos
- Backend: Rust + Axum
- Base de datos: SQLite
- Base de datos real: database/nebripop.db
- Backend local: http://127.0.0.1:3000
- Frontend local: http://127.0.0.1:8081
- Control de versiones: Git + GitHub

Objetivo principal de la Fase 9:
Implementar una mensajería básica entre usuarios, permitiendo enviar mensajes relacionados con un producto, consultar conversaciones y ver mensajes recibidos o enviados.

Importante:
Esta fase debe centrarse únicamente en mensajería básica.

## 4. RustBackend

RustBackend será el agente principal de implementación backend de esta fase.

Debe trabajar en:

backend/

Debe implementar endpoints para mensajería básica.

Endpoints obligatorios:

### POST /messages

Debe enviar un mensaje relacionado con un producto.

Body esperado:

{
  "product_id": 1,
  "sender_id": 2,
  "receiver_id": 1,
  "content": "Hola, ¿sigue disponible?"
}

Validaciones:
- product_id debe existir.
- sender_id debe existir.
- receiver_id debe existir.
- sender_id y receiver_id no deberían ser iguales.
- content no puede estar vacío.
- content debe tener una longitud máxima razonable, por ejemplo 1000 caracteres.
- No guardar en JSON.
- Guardar en SQLite.

Respuesta esperada:

{
  "id": 1,
  "product_id": 1,
  "sender_id": 2,
  "receiver_id": 1,
  "content": "Hola, ¿sigue disponible?",
  "created_at": "2026-05-24T10:00:00",
  "is_read": false
}

### GET /users/:id/messages

Debe devolver mensajes recibidos por un usuario.

Ejemplo:

GET /users/1/messages

Respuesta esperada:

[
  {
    "id": 1,
    "product_id": 1,
    "sender_id": 2,
    "receiver_id": 1,
    "content": "Hola, ¿sigue disponible?",
    "created_at": "2026-05-24T10:00:00",
    "is_read": false
  }
]

### GET /users/:id/messages/sent

Debe devolver mensajes enviados por un usuario.

Ejemplo:

GET /users/2/messages/sent

### GET /products/:product_id/messages?user_id=1

Debe devolver la conversación relacionada con un producto donde participa el usuario indicado.

Ejemplo:

GET /products/1/messages?user_id=1

Debe devolver mensajes donde:

- product_id = 1
- sender_id = user_id OR receiver_id = user_id

Ordenados por created_at ascendente.

### PATCH /messages/:id/read

Debe marcar un mensaje como leído.

Body esperado:

{
  "user_id": 1
}

Validaciones:
- El mensaje debe existir.
- Solo el receiver_id debería poder marcarlo como leído.
- Si user_id no coincide con receiver_id, devolver error.

Respuesta esperada:

{
  "message": "Mensaje marcado como leído",
  "id": 1
}

Reglas backend obligatorias:

- Usar SQLite.
- No usar JSON como persistencia.
- Usar consultas parametrizadas.
- Validar sender_id, receiver_id y product_id.
- Validar content.
- No exponer password_hash.
- No implementar WebSockets.
- No implementar chat en tiempo real.
- No modificar login salvo que sea imprescindible.
- Mantener compatibilidad con productos, usuarios y favoritos.
